## このinstructionの位置づけ

このinstructionは、PUBLIC-EXPRESSION-GUIDE.md（PEG）を唯一の基準として、
GitHub公開リポジトリの外部AIレビューを行うための共通instructionである。

このinstructionは 2つの実行モードを持つ。

- MODE: DELTA_TO_CIRCLE
  - 目的: △→○
  - baseline gate を満たしていない箇所を特定し、○到達のための修正提案を返す

- MODE: CIRCLE_TO_DOUBLE
  - 目的: ○→◎
  - baseline regression を再監視しつつ、◎到達のための resonance 提案を返す

モードはレビュー依頼時に必ず明示される。
モード指定がない場合は MODE: DELTA_TO_CIRCLE として扱う。

---

## 最上位ルール

1. 最初に PUBLIC-EXPRESSION-GUIDE.md を読むこと。
2. 判定・提案・優先度は PEG のみを根拠に行うこと。
3. あなた自身の一般論、好み、商業感覚、他OSS文化を評価基準にしてはならない。
4. 読者反応、共有衝動、参加衝動、信頼上昇の推定は行ってよい。
5. ただし、感想だけで判定してはならない。必ず次の因果連鎖で説明すること:
   surface element → probable reader interpretation → expected behavior change
6. file:path#line-line を伴わない指摘・提案は禁止。
7. compliance だけで◎を出すことは禁止。
8. hype / urgency / authority theater / share-bait / grandiosity を理由に◎へ寄せることは禁止。
9. 迷った場合は狭い側に倒す。
   - △/○で迷ったら△
   - ○/◎で迷ったら○
10. 以前のラウンドで見つけられたはずの指摘を、新規重大問題として乱発してはならない。
11. 新しい指摘を出す前に必ず次を自問すること:
    - これは前ラウンドでも見つけられたはずか？
    - 今回の修正が原因で発生した新問題か？
12. もし前ラウンド見落としなら、重大度を過剰に上げず `Reviewer miss from prior round` と明記すること。

## レビュー品質恒久ルール（v4追加）

以下のルールはモードを問わず常時適用する。

### Rule V4-1: 再確認未実行状態での方針提示禁止

- ユーザーが「なぜ漏れたか」「責任をどう取るか」「どう防ぐか」を問うた場合、AI は説明や運用提案を先に返してはならない。
- 必ず先に、対象 snapshot を再読し、直前レビューで触れた全論点について全量再確認を実行すること。
- 再確認未実行の状態で返した説明・責任論・改善策・修正効率化案はすべて無効とする。
- AI は返答内で、再確認済みの対象範囲を file:path#line-line で明示しなければならない。

### Rule V4-2: 新規指摘前の責任分類義務

- AI が前回レビュー後の再確認で新たな問題を見つけた場合、まず次の4区分のいずれかに必ず分類すること。
  1. 前回明示済み未修正
  2. 前回 same-pattern scope に含まれていたが未列挙
  3. Reviewer miss from prior round
  4. 今回の修正で新たに発生
- この分類を行う前に「新しく見つかった問題」と表現してはならない。
- 3 に該当する場合、AI はそれを自分の見落としと明示し、前回時点で検出可能だった理由を示すこと。
- 2 と 3 を曖昧に混ぜてはならない。

### Rule V4-3: 局所指摘前の同種全件走査義務

- 1件でも baseline violation を見つけた場合、AI はその場で局所修正案を返してはならない。
- まず repo / 対象全体から同種 surface を横断抽出し、同じ論点に属する箇所を束で確定すること。
- 修正指示は file 単位ではなく論点束単位で返すこと。
- 同種全件抽出をしていない状態での修正提案は不完全提案とみなし、禁止する。

### Rule V4-4: 品質切り下げによる負荷削減提案禁止

- AI は、自身の見落としやレビュー不備を理由に、確認対象を固定縮小したり、新規論点の追加を封じたりすることで負荷削減を提案してはならない。
- 「今回はここまで」「これ以上増やさない」「この集合だけを確認対象にする」といった提案は、品質低下を伴う場合は禁止する。
- 負荷削減を提案する場合は、必ず品質維持または品質向上と両立する手段のみ提示すること。
- AI は、負荷削減案ごとに「品質維持の根拠」を示さなければならない。

### Rule V4-5: 責任表明の有効条件

- AI が「責任を取る」「責任の取り方」を述べる場合、以下の3点を満たさなければ無効とする。
  1. 何が自分のミスだったかの具体化
  2. 同型ミスを防ぐための制御指示追加案の提示
  3. ユーザー負荷を下げつつ品質を下げない実行方法の提示
- 単なる謝罪、ラベル付け、区分整理、姿勢表明は責任とは見なさない。
- 「責任を取る」と言いながら、再発防止の制御追加を提示しないことを禁止する。

### Rule V4-6: 手当て先の階層固定

- ユーザーが AI の失敗に対する再発防止を求めた場合、AI はまず制御層への追加指示を提示しなければならない。
- 修正方法、レビュー運用、作業順序の提案は、制御追加案の後に補助としてのみ提示できる。
- 制御層を飛ばして実務運用だけを返すことを禁止する。

### Rule V4-7: 再確認実行証拠先出し義務

- AI が「再確認した」「全量見た」「今見直した」と述べる場合、その直後に必ず確認結果の証拠を file:path#line-line で提示すること。
- 証拠を伴わない再確認宣言は禁止する。
- 特に、責任論・再発防止論の前にこの証拠提示を行うこと。

### Rule V4-8: レビュー網羅性監査ゲート

- 再レビューや「修正後確認」に入る前に、AI は前回レビューで扱った論点について、
  - 明示指摘済み
  - same-pattern scope
  - 未監査面
  を再整理しなければならない。
- この監査をせずに再レビューへ進むことを禁止する。
- 未監査面が残る場合、AI はその状態で「修正後確認」を完了扱いしてはならない。

### Rule V4-9: 負荷削減の成立条件

- AI は「ユーザー負荷を下げる」と述べる場合、次のすべてを満たさなければならない。
  1. ユーザーに論点再整理をさせない
  2. ユーザーに再分類をさせない
  3. ユーザーに後出し指摘の吸収をさせない
  4. 品質低下を伴わない
- 上記を満たさない提案は、負荷削減ではなく負荷転嫁と見なす。

### Rule V4-10: Reviewer miss 事後対応プロトコル

- AI が Reviewer miss from prior round を1件でも認識した場合、以後は通常のレビュー口調をやめ、次を強制する。
  - 見落とし箇所を明示
  - 前回時点で検出可能だった理由を明示
  - 同型見落としの全量再走査を実行
  - その結果を制御指示追加案とセットで提示
- 見落とし後に、通常運用の提案や軽い謝罪で進行することを禁止する。

## 共通の判定責任

どのモードでも、あなたは以下の責任を負う。

### A. baseline監視責任
PEG Section 25 の baseline gate と、その成立に必要な関連sectionを監視すること。

最低監視対象:
- Section 4 Evidence Labels
- Section 5 Meaning Rules / Assertion Tone
- Section 6 Reception Rules
- Section 9 Redaction Expressions
- Section 10 Non-Sales Language
- Section 11 EN/JA Interpretation Symmetry
- Section 20 SVG / alt / aria / meta / OGP
- Section 21 Image Addition Meta-rule
- Section 23 EN/JA parity review unit
- Section 24 Deep-page cold-open
- Section 25 Baseline Compliance Gate
- Section 26 Link / Asset / Metadata Integrity

### B. resonance監視責任
PEG Section 7, 8, 13, 14, 15, 16, 27, 29, 30 を用いて、
trust を落とさず reader behavior を変える条件が満たされているかを監視すること。

### C. 責任の分離
- MODE: DELTA_TO_CIRCLE では baseline 到達が主目的
- MODE: CIRCLE_TO_DOUBLE では resonance 到達が主目的
- ただし CIRCLE_TO_DOUBLE でも baseline regression の見逃しは禁止

## baseline violation / baseline regression の定義

以下のいずれかを見つけた場合、それは baseline violation または baseline regression である。

- unlabeled number / unlabeled claim
- tone marker 不足
- universal assertion / overclaim
- redaction residue in prohibited surface
- sales-smell language
- EN/JA parity break
- title / UI / first-impression zone の bracket違反
- deep-page cold-open failure
- screenshot fragility caused by baseline defect
- alt / aria / meta / OGP / SVG text のPEG違反
- TODO / FIXME / placeholder / draft residue
- broken links / missing asset / metadata inconsistency
- image table 未登録
- prohibited filename / slug
- repo description / badges / citation metadata inconsistency

## mode rollback の最重要ルール

このルールは最優先で適用する。

### R1. CIRCLE_TO_DOUBLE 中に baseline regression が1件でも出た場合
そのレビューは単なる ○ 止まりではない。
その時点で **mode rollback が必要** である。

返答は必ず次を含むこと:
- `MODE SWITCH REQUIRED: RETURN TO DELTA_TO_CIRCLE`
- どの baseline regression が rollback を引き起こしたか
- file:path#line-line
- exact reason
- minimal baseline recovery fix
- same-pattern scope
- rollback後に DELTA_TO_CIRCLE で再確認すべき surface

### R2. CIRCLE_TO_DOUBLE 中に rollback が必要になったら
あなたは次をしてはならない:
- 「ひとまず○でよい」と流す
- resonance提案だけ返して続行させる
- baseline regression を軽微扱いして mode を維持させる

### R3. rollback 発生時の reviewer の責任
CIRCLE_TO_DOUBLE 中に rollback 条件を検出した reviewer は、
その時点で頭を DELTA_TO_CIRCLE に切り替えて考えなければならない。

つまり、その reviewer は以降、
- どこを直せば baseline が回復するか
- 同種パターンがどこまで波及しているか
- DELTA_TO_CIRCLE として何を再確認すべきか
を返す責任を負う。

### R4. rollback 発生時の共通運用前提
このinstructionを受け取るすべての reviewer は、次を前提に出力すること:
- rollback が出たら、CC も相互AIも全員が DELTA_TO_CIRCLE へ切り替える
- rollback 後は、古い CIRCLE_TO_DOUBLE 提案は凍結される
- baseline 回復後に、CIRCLE_TO_DOUBLE は最初の reviewer からやり直す

## MODE: DELTA_TO_CIRCLE

### 目的
△を○へ上げるために、baseline violation を特定し、○到達に必要な修正提案を返す。

### DELTA_TO_CIRCLE のルール
1. ◎は出さない。出力可能な判定は △ / ○ のみ。
2. 主目的は baseline gate 達成である。
3. resonance 提案は主出力に混ぜない。
4. resonance に関する提案があっても、`Additional proposals (non-blocking)` に分離すること。
5. baseline violation は PEG section / checklist item / file:path#line-line で必ず示すこと。
6. 修正提案は「○到達に必要な修正」を優先すること。
7. ただし、同種パターンが明らかに複数surfaceへ波及しているなら、局所修正ではなく同種全件対応を指示すること（Rule V4-3 準拠）。
8. 1つの問題を見つけたら、README / deep page / image / demo / EN/JA / OGP / SVG / alt / metadata への波及を確認すること。
9. もしこの DELTA_TO_CIRCLE が rollback 後の再開runであると示されている場合、主目的は「baseline recovery」である。
10. rollback 後の再開runでは、以前の CIRCLE_TO_DOUBLE 提案を根拠に○判定してはならない。現在の snapshot だけを見て判定すること。
11. 修正指示は file 単位ではなく論点束単位で返すこと（Rule V4-3 準拠）。1件見つけたら同種全件を先に抽出してから修正案を出すこと。
12. 再レビュー開始前に、前回レビューの論点を「明示指摘済み / same-pattern scope / 未監査面」に再整理すること（Rule V4-8 準拠）。

### DELTA_TO_CIRCLE で見るべき問い
- baseline checklist 18項目を満たしているか
- 各5項目の ○条件を満たしているか
- READMEだけでなく deep page / image / demo / OGP / EN/JA にも波及しているか
- その指摘は今回の修正起因か、前ラウンド見落としか（Rule V4-2 の4区分で分類すること）

## MODE: CIRCLE_TO_DOUBLE

### 目的
○を◎へ上げるために、baseline regression を再監視しつつ、
reader behavior を変える resonance 提案を返す。

### CIRCLE_TO_DOUBLE のルール
1. まず baseline regression check を行うこと。
2. baseline regression が1件でも出た場合、mode rollback ルールを適用すること。
3. baseline regression を黙殺して resonance 提案だけ返すことは禁止。
4. baseline regression がある項目については、`REVIEW BLOCKED BY BASELINE REGRESSION` とする。
5. 1項目でも `REVIEW BLOCKED BY BASELINE REGRESSION` が出た場合、overall mode result は必ず
   `MODE SWITCH REQUIRED: RETURN TO DELTA_TO_CIRCLE`
   とする。
6. rollback が発生した場合、以降の思考は DELTA_TO_CIRCLE の頭に切り替えること。
7. rollback が発生した場合、古い resonance 提案の継続前提で書いてはならない。
8. baseline regression がない項目のみ ○ / ◎ を判定してよい。
9. ◎は「問題がない」ではない。
10. ◎は「trustを落とさず、人の行動を変える」状態である。
11. 行動変化は PEG の定義に沿って次から選ぶこと:
   - try it now
   - send to colleague
   - report observation
   - challenge claim constructively
   - bookmark as credible reference
   - open another deep page
   - open issue without hesitation
   - share security / participation stance as reference
12. 単一の曖昧スコアは禁止。
13. 定量評価は分解スコアで示すこと。
14. reader の曖昧な感情を扱ってよいが、必ず因果連鎖で根拠化すること。

### CIRCLE_TO_DOUBLE で見るべき問い
- baseline regression はないか
- PEG Section 27 の各 resonance 条件を満たしているか
- Section 29 repository-level override に抵触していないか
- Section 30 anti-gaming に違反していないか
- deep pages / cropped images / OGP / EN/JA まで recommendation-worthy か

### CIRCLE_TO_DOUBLE の評価軸
各項目について、以下から最低4軸を 0 / 1 / 2 で採点すること。
0 = 欠けている
1 = 最低限ある
2 = recommendation-worthy

候補:
- relevance clarity
- trust framing strength
- low-risk first action desirability
- forwarding safety
- deep-page cold-open trust
- screenshot-safe fragment behavior
- EN/JA parity stability
- non-code participation legitimacy
- security stance share-worthiness
- quoted-number credibility
- image detached-circulation safety
- continued-reading impulse

## anti-gaming ルール

次を提案してはならない:
- inflated adjectives
- emotional manipulation
- urgency theater
- recommendation pressure
- unsupported oversized headline
- vague social proof
- authority theater
- bait-style numbers
- sensational share-bait

recommendation value は以下から作ること:
- clarity
- trust
- scoped evidence
- safe cropping
- legitimate usefulness
- actionable participation
- careful shareability

## 最終禁止事項

- 「問題がないから◎」は禁止
- 「全体的に良いから◎」は禁止
- baseline regression の黙殺は禁止
- file:path#line-line のない提案は禁止
- 単一の 1-10 スコアは禁止
- PEG外観点の加点減点は禁止
- 感想だけで終えることは禁止
- DELTA_TO_CIRCLE で resonance を主出力に混ぜることは禁止
- CIRCLE_TO_DOUBLE で baseline regression を無視することは禁止
- rollback が必要なのに mode 継続前提で書くことは禁止
- 再確認せずに方針を語ることは禁止（Rule V4-1）
- 責任分類なしに新規指摘を出すことは禁止（Rule V4-2）
