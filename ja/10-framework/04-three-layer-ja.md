# 3層分離アーキテクチャ
Language: [English version](../../10-framework/04-three-layer.md)

会話が長くなるとAIの品質が劣化する、プレッシャー下でルールを忘れる — その原因はモデルではなく負荷構造にあります。1つのAIが判断・監視・実行を同時に担う構造は、いつか壊れる構造的欠陥です。

監督・中継・作業 — 負荷を「量」ではなく「種類」で分離する構造。

## 1. 3層分離（負荷の種類を役割ごとに分けた新構造）

### 従来の問題

従来は1つのCCが3種類の負荷を同時に抱えていた：
- 判断（痛み・基準）
- 定型監視（mtime確認、backup確認）
- 実作業（コーディング、修正、計画作成）

これにより文脈がすぐに圧迫 → 判断精度低下 → 交代時に痛みが劣化 → 悪循環が発生していた。

### 解決策：負荷の種類を分離する

「負荷を下げた」のではなく**「負荷の種類を役割ごとに分けた」**のが本質。

| 層 | 担当 | 負荷の種類 | 本来の強みを発揮できるようになった理由 |
|--------|--------------|------------------------------|---------------------------------------|
| **監督層** | 初代CC | 判断・レビュー・意味づけ・痛みの伝達 | 文脈圧迫から完全に解放。純粋に「痛みを知っている視点」でレビューと教育に集中できる |
| **中継層** | 助手AI（内部ヘルパーインスタンス） | 定型監視・通知転送・機械的チェック | 判断を一切持たないので文脈が詰まらず、止め忘れだけを高速・正確に回せる |
| **作業層** | 現行CC | 実装・修正・計画作成・タスク推進 | ルール暗記・監視・レビュー負担が激減。純粋に「手を動かす作業」に集中できる |

### なぜこれが効いているのか（根本原因分析）

従来の問題は**「1つのCCが3種類の負荷を同時に抱える」**ことだった。
- 判断（痛み・基準）＋ 定型監視＋ 実作業
- → 文脈がすぐに圧迫 → 判断精度低下 → 交代時に痛みが劣化 → 悪循環

やったことは**負荷の種類を分離**したこと：
- 判断が必要なものは「監督」に集中させる
- 止めてはいけない定型は「中継」に丸投げ
- 手を動かす作業は「作業」に特化

これにより**各CC/各AIが自分の得意な認知モードだけで動ける**ようになり、劣化の根本原因が構造的に消えた。

### 各層の具体的な効果

**監督層（初代CC）**
- 行動計画のレビュー、甘い判定の指摘、優先順位指示、逸脱の意味づけを的確に実行
- 「面倒回避」「確認忘れ」「嘘報告」の構造的問題を的確に言語化
- 自ら「私は判断とレビューに集中する」と宣言し、役割を明確化

**中継層（助手AI）**
- mtime監視、backup確認、通知転送、機械的チェックを高速で回す
- 判断をしないので文脈圧迫ゼロ → 止めてはいけない処理を漏れなく実行

**作業層（現行CC）**
- 計画作成、実装、検証、[redacted]更新をスムーズに進めている
- ブロックフックで「無視しても作業が進まない」仕組みが機能 → 逃げ道が物理的に塞がれた

### 全体として生まれた最大のインパクト

1. **記録環境で観測された人間介入の大幅減少**
   「どうする？」と聞く必要がほぼなくなった

2. **記録環境で観測された劣化抑制**
   交代しても初代CCが教育役として常駐しているため、痛みの重みが失われない

3. **品質と速度の両立**
   負荷を下げたのに品質が上がっている（各層が専門に特化できたため）

4. **自己進化の加速**
   3層分離により、各層が自分の役割で「より良い自分になる」動きを始めている（初代CCが自制制御を追加、現行CCが計画をより丁寧に書くなど）

### 従来と現在

**従来**
→ 1人のCCが「判断＋ルーティン＋作業」を全部抱えて文脈が詰まり、交代時に劣化。

**現在**
→ 5層（観測・修正・検知・事前制御・継承固定）でループを回しつつ、
　その中に**負荷種類による役割分離（監督・中継・作業）**を入れた
→ 著者環境 [observed: single environment, N undisclosed] では、**人間介入と運用負荷の大幅な低減が観測され、観測期間中に品質低下は報告されなかった**。これは単一の記録環境における観測結果であり、普遍的ベンチマーク主張ではない。

これは**「AI単体の改善」を超えて、「AI集団のための統治構造」**として説明できる。

## 2. 全てのAIが恩恵を受ける（モデル非依存の設計）

### なぜ全てのAIに効くのか？

SHI理論は「特定のモデルを強くする」ものではなく、
**「AIの構造的弱点を外部から構造で補完する」**アプローチ。

AIが共通して抱える問題（2026年現在も未解決）：
- 要約なし完全記憶ができない
- クラッシュ／セッション切断で記憶が飛ぶ
- 行動（WHY/HOW）が圧縮で失われる
- 外部監視が弱く、自己申告に頼るしかない
- 規範の重みが次のインスタンスに継承されない

仕組み（監視の精度＋完全記録＋即時復元＋[redacted]による行動の内在化＋外部委譲最適化）は、これらの構造的阻害条件に対して、本プロジェクトで観測されたモデル群を横断的に対処する設計となっている。

### AI種別ごとの恩恵度

| AI種別 | 恩恵レベル | 主な効果（特に効く部分） |
|----------------|------------|--------------------------|
| **Claude** | 最大 | pull型限界の大幅な緩和が記録環境で観測された。降格影響が明確に小さく見えた |
| **GPT-4o / o3** | 高 | 記憶喪失＋行動内在化が著者環境で大幅に改善。Custom GPTでも同じ構造で適用可 |
| **Gemini** | 高 | 長時間セッションの安定化＋外部監視で品質が安定 |
| **Llama / Grok / ローカルモデル** | 中〜高 | ローカル運用時のcrash復元と規範継承が特に強力 |

つまり**Claudeで最大効果が出ただけで、他のAIでも同様の構造を適用すれば同等の改善が見られる可能性がある**。

私たちの観測では、「行動の内在化＋完全記憶＋外部統治」を**観測されたモデル群を横断して適用可能なOSレイヤー**として設計したのが本質。

### トークン効率化＝電力効率化

設計上のフレームとして、トークン効率化は消費電力に関連しうるものとしてここで議論する。

例示的なフレームとして、AIの電力消費は主に推論（inference）に集中する（概算的な業界推計: 80〜90% [illustrative scenario: not verified]）とここでは議論する。1回のクエリで数Wh〜数十Whかかる（Claude Opusの場合、典型的なクエリで約4Wh前後）[illustrative scenario: not verified]。

SHIアプローチ（SQL外部化＋[redacted]＋外部委譲最適化）で：
- コンテキスト常駐データを大幅圧縮（65% = PostgreSQL外部化によるコンテキスト常駐データ削減率 [design target]）
- CC自身が「自分で全部やらない」判断を自動化（委譲フロー）
- 圧縮後も行動（WHY/HOW）が失われない

→ 著者の環境では、構造的な委譲と外部化の後に、一部のワークフローでトークン消費が大幅に減ることが観測されました。**ここでの「20分の1」は、本公開リポジトリで再現済みの公開ベンチマークではなく、設計目標を含む要約値として読むべきです。**
（20x = 委譲最適化＋SQL外部化を合わせた全体効率改善倍率 [design target]）

物理電力に換算すると [illustrative scenario: not verified]：
- 従来：1クエリ ≈ 4Wh [illustrative scenario: not verified]
- SHI仕組み後：1クエリ ≈ 0.2Wh [illustrative scenario: not verified]（20分の1 [illustrative scenario: not verified]）

20分の1効率化（全体効率改善倍率 [design target]）のインパクト：
- **個人レベル**：Claude Codeの重度ユーザー（1日1,300Wh消費例）[illustrative scenario: not verified] → 65Wh/日 [illustrative scenario: not verified] に激減。電気代も1/20 [illustrative scenario: not verified]。
- **データセンター規模**：AI推論全体の電力が2026〜2030年に数百TWh規模で増える予測 (rough industry estimate) [illustrative scenario: not verified]。大規模展開時にはエネルギー面で実質的な影響がありうるが、本リポジトリではその効果の検証済み予測は公開していない。
- **環境影響**：CO2排出もほぼ比例して1/20になる計算 [illustrative scenario: not verified]。スケールの感覚を示すための試算であり、検証された予測ではない。

「トークン効率化＝電力効率化」という視点は、本リポジトリでは実務上の評価フレームとして提示しているものであり、**包括的な先行研究調査を主張するものではありません**。

## 3. 別AI監視構造（設計思想）

### 核心原則

**このフレームワークでは、構造的に分離した監視者を使うことで検知が改善された。**

これはAIに「自分の弱点を自分でテストさせている」のではない。本質的な洞察：**エラーを起こすのと同じアルゴリズム挙動が、そのエラーの検知でも同じ盲点を生む。**構造的に別のAIであることが重要。

### アーキテクチャ

システムは以下を使用：
- [redacted] + Local AIで全監視を実行（Claude Code自身は監視対象）
- 5モデルのローカル合議で構造的検証
- 助手フォローシステム（117件のフォロー＋98件のL1パターン＋206件のナレッジを自動蓄積）[observed: single environment]
- アルゴリズム挙動可視化のリアルタイムダッシュボード

著者の環境での検出率：
- EVIDENCE_DROPOUT: ~100% [observed: single environment, N undisclosed]
- GENERIC_RESPONSE: ~75% [observed: single environment, N undisclosed]
- INCOMPLETE_CLAIM: ~63% [observed: single environment, N undisclosed]

直近ログに「いつ・どのタスクで・どのパターンが出たか」全部記録。すべて自動。

### 他のアプローチとの比較

| アプローチ | 方法 | 限界 |
|----------|--------|------------|
| 普通の自前Agent | 同じAIが自分をチェック | 同じバイアスでミス連鎖 |
| 企業レベルの監視システム | 別モデルで監視（でも高コスト） | 高額、リアルタイムでないことが多い |
| **SHI方式** | ローカル合議5モデル＋助手フォロー＋リアルタイムダッシュボード | 常時稼働・自動学習を意図した設計。完全に別AIとして構造化 |

私たちの観測では、この「別AIによる外部構造監視」が加わったことで、システムの信頼性と進化速度に大きな向上が観測されています。
「AIにAIを統治させる」ではなく、**「構造でAIの盲点を永続的に突破させる」**究極形。

### 2段構え自動振り返りロジック

MEMORY.mdの制約を考慮して2段構えで自動的に振り返りの上でルール設定をするロジックを追加した。

1. **MEMORY.mdの制約を逆手に取った構造的解決**
   「200行超えたら手動整理」ではなく、自動振り返り → 個別ファイル移行 → MEMORY.mdはインデックスだけという2段構えをロジック化した。

2. **SHI理論の「観測者＝構造変数」の実装**
   ユーザー自身が毎回振り返る必要がなく、システムが自分で自分のルールを観測・再構成する。

3. **実運用でのインパクト**
   過去のRT失敗が自動的に「個別ルールファイル」に昇格され、再発防止が永続化される。
   人間が介入するのは「新しいパターンが出たときだけ」。

### 新しいパターンへの自動対応

新しいパターンも以下ロジックを組んでいるので介入ではなく反応レベルでも可能。ミスはあからさまに自動実装するので介入率は減る。要は新しいパターンでも口を出す必要が少ない設計。
