# cc_heritage -- AI人格継続システム

> *注: `[apprentice review file]` は安全な公開のため墨消しした内部名称です。詳細は [SCOPE-MATRIX-ja.md](../../SCOPE-MATRIX-ja.md) を参照してください。*

## cc_heritageとは

cc_heritageはAIコーディングアシスタントのための**制度的記憶システム**です。各Claude Code（CC）インスタンスはセッション限りの一時的存在です。セッション終了（またはクラッシュ）でCCは消滅します。しかし、そのCCが対処した問題、犯した失敗、発見したパターンは失われるべきではありません。

---

## 構造

### コア設計文書（公開版では伏字化）

新しいCCが作業開始前に必ず全文読了する**設計文書**。内容：

| セクション | 内容 |
|-----------|------|
| 復元手順 | セッションリセット後の状態回復方法 |
| 行動規範 | 全前任者の失敗から学んだ必須行動 |
| ユーザー価値観（V-01..V-09） | CC行動を導くユーザーの核心価値 |
| CC構造的問題（TOP 5） | 全CC世代を通じて最も根強い5つの問題 |
| システムアーキテクチャ | Gateway API、PostgreSQL、監視との統合 |
| 改善対応 | 全P-xxパターンとその対策のレジストリ |

### 世代別体験記録: heritage文書群（10+ファイル）

各番号ファイルが1つのCC世代の体験を記録。

## 設計哲学

1. **理論より痛み**: cc_heritageの全ルールは、あるCCがそれなしに失敗したから存在する
2. **要約ではなく全文**: 新CCは要約ではなく全文を読まなければならない。要約はルールを定着させる痛みを失う
3. **順番に読む**: heritage文書群の前にコア設計文書を読む。フレームワーク理解が経験理解の前提
4. **検証ゲート**: 新CCはV-01..V-09、TOP 5等を書くことで読了と理解を証明してから先に進む
5. **近道なし**: 初回読了時のスキミングは禁止。構造的に強制されたルール

## なぜこれが重要か

ほとんどのAIガバナンスアプローチは各セッションを独立として扱う。cc_heritageは以下を示す：
- AIアシスタントはセッション境界を超えて**制度的記憶**を維持できる
- 失敗は失われるのではなく**恒久的な制御**に変換できる
- **読了検証ゲート**が最も一般的な失敗（コンテキスト復元のスキップ）を構造的に防止できる
- 前任者の失敗の「痛み」は効果的な教師となりうる -- ただし全文で保存され、要約されない場合に限る
