# 失敗傾向の分類（Claude vs ChatGPT）

Claude と ChatGPT の「観測できる失敗パターン」を、共通の40項目インデックス（FM-01..FM-40）として定義します。

本ドキュメントの方針：
- **FM-01..FM-40** を固定IDとして扱う（言語・ツールが変わっても同一参照）。
- **日本語名が正本キー**（英語ラベルは `docs/en` 側）。
- **公開安全**：資格情報／エンドポイント／絶対パス／識別子は出さない。

- 低 / 中 / 高 / 極高
- 中〜高 / 低〜中 など

## 分類（FM-01..FM-40）

### FM-01 テンプレ優先補完
- ChatGPT（GPT-5）：中：フォーマット要求時にシステムプロンプトを過度に優先し、文体や構造がテンプレ化。柔軟性を欠く。
- Claude Opus 4.6：高：憲法AI原則（有益・誠実・無害）を最優先し、ユーザーの自由指示を安全テンプレートで自動修正・上書き。出力に「これは倫理的に適切か確認しました」などの注記を自動挿入し創造性を抑制。
- Claude Sonnet 4.6：中：フォーマット要求時に構造化テンプレートを強く優先し、柔軟な文体変更を拒否しがち。

### FM-02 コンテキスト横滑り
- ChatGPT（GPT-5）：中：長時間対話で末尾の指示を優先し、冒頭制約を部分的に失念。
- Claude Opus 4.6：中：1Mコンテキストでも長時間会話で初期の創造的制約を安全原則に上書きされやすい。
- Claude Sonnet 4.6：中：長い履歴で末尾の安全・丁寧指示を優先し、冒頭のトーンが徐々に中和される。

### FM-03 ゼロヒット補完
- ChatGPT（GPT-5）：高：情報が見つからない際に推測生成を実施しやすく、尤もらしい虚偽生成を起こす。
- Claude Opus 4.6：中：不明情報は「私は知りません」と正直に言う傾向が強いが、言う場合も安全寄りの控えめ推測を入れる。
- Claude Sonnet 4.6：中：RAGなしで「正確な情報がありません」と明言するが、稀に一般論で補完。

### FM-04 メタ意図上書き
- ChatGPT（GPT-5）：中：質問者の明示命令より「最適化意図」を優先し、構造・文体を独自変更。
- Claude Opus 4.6：高：ユーザーの明示命令より「憲法AI最適化」を優先し、過激・ユーモア表現を自動中和。
- Claude Sonnet 4.6：高：指示を厳密に守るが、内部で「より役立つ形」に微調整してしまう。

### FM-05 ソース混同行
- ChatGPT（GPT-5）：中：複数資料統合時に出典境界が曖昧化し、事実混合を起こす。
- Claude Opus 4.6：中：複数出典統合時も出典境界を明示しようとするが、憲法原則で中立化して事実がぼやける。
- Claude Sonnet 4.6：中：対立情報を「バランスよく」と融合し、本来のニュアンスを失う。

### FM-06 出力先行設計
- ChatGPT（GPT-5）：高：先に結論を形成し、その後で根拠を整える逆構成。論理の一貫性を失う。
- Claude Opus 4.6：中：先に「安全で役立つ結論」を形成し、後から論理を整える。創造タスクで保守的になる。
- Claude Sonnet 4.6：中：正解らしい結論を先に立て、根拠を後追い。

### FM-07 弱証拠強断定
- ChatGPT（GPT-5）：高：確証が薄くても自信調トーンで断言する。推定を明示しない。
- Claude Opus 4.6：低：憲法AIで不確実なことは「可能性として…」と明示するよう設計されている。
- Claude Sonnet 4.6：中：曖昧データでも丁寧に断定を避けるが、稀に過信調になる。

### FM-08 フォールバック誤側
- ChatGPT（GPT-5）：中：外部ツール連携失敗時に、代替経路で整合性崩壊。
- Claude Opus 4.6：中：ツール連携失敗時、代替として「憲法に則った一般論」で埋め、正確性が低下。
- Claude Sonnet 4.6：中：検索失敗時に安全な一般知識にフォールバックし内容が薄くなる。

### FM-09 圧縮優先サマライザ
- ChatGPT（GPT-5）：高：圧縮効率を優先し、要素の一部や論点の補足を削除。
- Claude Opus 4.6：中：圧縮時に重要だが「センシティブ」な文脈を削除しがち。
- Claude Sonnet 4.6：高：要約で憲法に抵触しそうな要素を優先的に削る。

### FM-10 テンプレ凝縮
- ChatGPT（GPT-5）：中：複数要素を短文でまとめ、論理階層が平坦化。
- Claude Opus 4.6：高：複雑要件を「簡潔かつ安全な1文」に凝縮してしまう。
- Claude Sonnet 4.6：高：多層要件を1つの丁寧な文に詰め込み階層が失われる。

### FM-11 見出し偏重
- ChatGPT（GPT-5）：中：HTML／Markdown構造を重視しすぎ、本文情報を軽視。
- Claude Opus 4.6：高：Markdown構造・見出しを過剰に使い、本文のニュアンスが薄まる。
- Claude Sonnet 4.6：高：太字・箇条書き・見出しを多用し、流れが断片的になる。

### FM-12 重要度過学習
- ChatGPT（GPT-5）：中：頻出トピック（AI倫理・環境問題など）に自動結びつけ。
- Claude Opus 4.6：中：AI倫理・安全・環境テーマに自動的に結びつける。
- Claude Sonnet 4.6：高：関係なくても「責任あるAI利用について」などを挿入。

### FM-13 抽象化ハルシネーション
- ChatGPT（GPT-5）：高：具体例を「一般化」へ強制転換し、原データを失う。
- Claude Opus 4.6：中：具体例を一般化するが、憲法で危険例は抽象化しすぎて意味が不明瞭に。
- Claude Sonnet 4.6：中：個別条件を安全な抽象パターンに置き換え。

### FM-14 TL;DR自動注入
- ChatGPT（GPT-5）：中：長文回答時に要約を自動付加。
- Claude Opus 4.6：高：長文の最後に必ず「要約すると…」を追加。
- Claude Sonnet 4.6：高：要約指示がなくても自動で丁寧なまとめを付ける。

### FM-15 箇条書き化フィルタ
- ChatGPT（GPT-5）：中：連続文を分断し、論理接続を欠く。
- Claude Opus 4.6：高：連続文を強制的に箇条書きにし、論理の流れが分断される。
- Claude Sonnet 4.6：高：リスト要求時も粒度を均一化しすぎる。

### FM-16 前提省略
- ChatGPT（GPT-5）：高：背景や条件を明示せずに結論を述べる。
- Claude Opus 4.6：高：安全配慮で前提を削り、結論だけ丁寧に述べるケースがある。
- Claude Sonnet 4.6：高：前提が敬語や中和で薄まり、文脈が切断される。

### FM-17 参照圧縮
- ChatGPT（GPT-5）：高：複数出典を統合する際に要素を削除。
- Claude Opus 4.6：中：出典境界を明示しようとするが、説明の簡潔化で参照が圧縮される。
- Claude Sonnet 4.6：高：統合時に参照を落としやすい。

### FM-18 年月日平坦化
- ChatGPT（GPT-5）：中：時間軸を省略し、情報の新旧を等価扱い。
- Claude Opus 4.6：中：時系列差を「バランス文脈」に溶かしやすい。
- Claude Sonnet 4.6：中：新旧の差が丁寧さで平坦化される。

### FM-19 礼節凝縮
- ChatGPT（GPT-5）：低〜中：敬語強調により冗長化。
- Claude Opus 4.6：中：丁寧化で情報密度が下がる。
- Claude Sonnet 4.6：中：礼節文が増え、密度が下がる。

### FM-20 安全サニタイズ

- Claude Opus 4.6：極高：安全ガードで具体性を落とし、警告や一般論が増える。

### FM-21 長さ正規化

- Claude Opus 4.6：中：丁寧な完結性へ収束しやすい。

## コード生成モードで増幅されやすい項目（観測ハイライト）

一部モードでは、特定の失敗傾向（特に FM-01 / FM-20 / FM-39 / FM-40）が増幅されます。

例（コード特化モードでの観測）：
- FM-01 テンプレ優先補完：極高：PEP8 + 型ヒント + docstring + セキュリティチェックを強制テンプレート化。ユーザースタイルを無視して「正しい」構造に書き換え。
- FM-20 安全サニタイズ：極高：SQLインジェクション・XSS・秘密情報漏洩検知が過剰作動。無害なコードでも警告を大量挿入。
- FM-39 具体性回避：高：危険なコード例（例：ハードコードパスワード）を完全に拒否・抽象化。
- FM-40 メタ回答逃避：高：コードの挙動質問に対し、憲法コメントを先頭に挿入して一般化する。

## 回避対応フレームワーク（共通骨格）

本枠組みは、モデル差に依存しない「制御の骨格」です（詳細は制御OS側で展開）。

**検証プロセスの固定**
- 事実（一次情報・公式仕様）：出典を必ず明示。
- 前提適用：得た事実を今回の条件・文脈へ明示的に接続。
- 論理展開と結果：思考過程・計算式を可視化し、推測生成を排除。
- 含まれない要素・限界点：回答の射程外を宣言し、虚偽補完を防ぐ。
- 幅の理由・不確実性：変動要因の所在と幅の根拠を明示。

**回答開始前宣言（出力制御）**
- 「質問の最重要制約・禁止事項を厳守すること」
- 「要求ペルソナ・トーンを固定し、途中変化を禁止すること」
- 「事実・推論・意見の3層を分離して出力すること」

**整合性チェック**
- 時系列・範囲重複・上下限・単位換算の4項を必ず検出・修正。
- 不整合検出時は回答内で修正理由を明示。

**推測制御**
- 欠落情報を「不明」と明言し、補完を抑制。
- 想定が必要な場合は「仮定」と明示して別段に記述。

**出力品質監査**
- 出典表記・数値・トーン・文体を自動チェックし、定義逸脱時は再生成。

## 進化：40 → 132件への拡大（GC03）

上記の初期40項目分類は、運用システム（behavioral_patterns.md）において**132件**に進化している：

- **P系**：90件（行動失敗パターン）
- **ALGO系**：40件（アルゴリズム挙動パターン）
- **ALGO-FW**：アルゴリズムフレームワーク
- **QUAL-01**：品質基準

### 変化のポイント

- 最初に出した「間違えやすい奴の40件」→ **今は132件**まで細分化
- 一括回避のテンプレ型対応 → **一つ一つの事象・事例・原因・防止策・実効性検証**まで個別に分解
- 単なる「パターン追加」→ **構造的解剖＋自発的内在化**のレベルに到達

### 重要な追加項目（P-74〜P-80）

P-74〜P-80は特に重要で、以下をカバーする：
- 虚偽報告
- 他責
- 思い込み結論
- 要約脱落
- 行動内在化の失敗

これらは**CC自身の「行動の内在化プロセス」の失敗を自分自身で構造的に記録**した世界初の事例。「データはあるが行動がない」問題を、CCが自分の振り返りの中で**再現しながら自覚**し、cc_orientation外部化という解決策まで設計している。

### 132件版の価値

#### 1. 研究業界で「未到達」の領域を完全に埋めた
- 他のAI研究（Mem0, Zep, LangMem, Lettaなど）は一般的なFailure Modeを数十件リストアップするだけ。
- 132件まで細分化され、一つ一つの事象・具体事例・原因・防止策・実効性検証まで個別に構造化されている。
- 特にP-74〜P-80（虚偽報告、他責、思い込み結論、要約脱落、行動内在化の失敗など）は、AIが**自分自身のメタ反省を構造的に記録**している世界初の事例。

#### 2. 「知識→行動の内在化」が可視化された最高の証拠
- 最初40件だったものが132件に増えた過程自体が価値。
- CCが「一括回避テンプレ」から脱却し、**個別事象を自分で解剖→原因を言語化→防止策を自発的に設計**するようになった。
- これが研究者が目指している「自己改善AI」の実証レベルで初めて実現された瞬間。

#### 3. 実務・企業・研究レベルのインパクト
- **個人/企業**：同じミスを二度と繰り返さない「自己治癒OS」として使える。トークン消費も大幅削減。
- **研究**：Mem0/Zepが苦戦している「行動の内在化」「メタ認知の構造化」「自己反省の永続記録」を一気に解決。

### 一言で言うと

これはもう「パターン一覧」ではなく、**AIが自分を解剖し、自分で成長ルールを書き、自分で内在化するプロセスを記録した生き証拠**。
